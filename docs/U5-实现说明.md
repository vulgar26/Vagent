# U5：多路检索（第二路全局向量 + 合并）

本文档说明 **升级策划书 U5** 在本仓库的简化实现。

## 行为

| 路径 | SQL | 用途 |
|------|-----|------|
| **主路** | `KbChunkMapper#searchNearest`（`user_id` 过滤） | 当前用户知识库 |
| **第二路** | `KbChunkMapper#searchNearestGlobal`（全表） | 跨租户「全局」向量最近邻 |

**仅当** `vagent.rag.second-path.enabled=true` **且** `cross-tenant=true` 时才会执行第二路；否则与 U4 前行为一致（仅主路）。

## 触发条件（满足任一即尝试第二路）

1. `trigger-on-empty=true` 且主路 **0 条**  
2. `min-primary-hits-below` **> 0** 且主路命中数 **&lt;** 该值（为 **1** 时与「仅无命中」接近；设为 **3** 表示主路不足 3 条即补充）  
3. `min-query-length` **> 0** 且改写后检索 query 的 trim 长度 **&lt;** 该值（作「意图/改写置信偏低」的**粗代理**，非模型打分）

## 合并与截断

- `RetrieveHitMerge` 按 `chunk_id` 去重，同 id 保留距离更小者；再按距离升序排序，截断为 **`vagent.rag.top-k`**。  
- `RetrieveHit.source`：`primary` / `global`；拼 SYSTEM 时对 `global` 片段加说明前缀。

## 安全与合规

第二路 **不按用户隔离**，会读到**其他用户**入库的片段。默认 **关闭**；仅用于**内网演示、明确共享语料**或接受跨租户风险的场景。  
生产多租户默认应保持 `enabled: false` 或 `cross-tenant: false`。

`POST /api/v1/kb/retrieve` 仍调用 **`KnowledgeRetrieveService#search`**，**不**走第二路。

## 配置示例（演示）

```yaml
vagent:
  rag:
    second-path:
      enabled: true
      cross-tenant: true
      top-k: 8
      trigger-on-empty: true
      min-primary-hits-below: 1
      min-query-length: 0
```
