# 工程折中与生产对照（决策用）

本文档把仓库里**为教学/演示做的简化**、**为省成本做的最小改动**，与**常见生产实践**对照列出，便于你逐项决定「保持 / 排期改 / 仅文档说明」。

> 与策划书差异的「产品层」取舍仍以 [DECISIONS.md](DECISIONS.md) 为准；本文偏**工程与安全**。

---

## 1. 教学与演示向简化

| 领域 | 当前做法 | 常见生产做法 | 是否建议改 |
|------|----------|--------------|------------|
| **LLM** | 默认 `noop`；可选 `fake-stream` / `dashscope` | 生产固定真实供应商；无 Key 用 profile 隔离而非默认 noop | 可选：默认 `dashscope` + 无 Key 时启动失败或单独 `dev` profile |
| **Embedding** | 默认 `hash`；可选 `dashscope` | 生产用真实嵌入；hash 仅测试 | 保持 hash 为默认可接受；生产配置强制校验 |
| **空检索** | 默认 `empty-hits-behavior: allow-llm` | 常选 `no-llm` 或产品可配置，避免无依据胡编 | 看产品：对齐 §3 可改默认 `no-llm` |
| **意图 / 改写** | 规则（前缀、长度、`tool=` 等） | 小模型/分类器或 LLM 结构化输出 | 演示保持；商用再迭代 |
| **记忆** | 仅 `messages` 最近 N 条，无摘要表 | 摘要表、向量记忆、TTL | 需求驱动 |
| **MCP** | 默认关；Debug API；主链路白名单 + 少量参数收敛 | 同上思路常见；生产加审计、配额、schema 校验 | 扩展按工具逐个 schema 化 |
| **前端** | `static/` 单页演示 | 独立前端工程、构建流水线、CDN | 演示足够；产品化再拆仓 |
| **可观测** | traceId + 少量 Timer | 分布式追踪、统一日志、SLO 告警 | 按需加 OpenTelemetry 等 |
| **测试** | H2 模拟 PG；M2 类默认排除 Surefire | 关键路径 PG/Testcontainers CI | 可把 M2 纳入夜间流水线 |
| **Actuator** | 暴露 `health,info,metrics` | 生产常限流、鉴权、网络隔离 | 收紧暴露项与网段 |
| **CSRF** | 关闭（无 Cookie 会话） | JWT 纯 API 常见关 CSRF | 若加 Cookie 登录需重评 |

## 2. 为速度/成本做的「最小改动」（技术债向）

| 领域 | 当前做法 | 常见更优做法 | 是否建议改 |
|------|----------|--------------|------------|
| **用户 ID 类型** | Flyway **V3** 起列为 **`uuid`**；Java 仍为 `String` + `UuidStringTypeHandler`；`UserIdFormats.canonical(UUID)` 与 `sub` 对齐 | 可进一步把实体改为 `java.util.UUID` 减少字符串 | **已做 DB+映射**；实体换 `UUID` 可再排期 |
| **历史：CHAR(36)** | 已改为 VARCHAR（曾用 CHAR 致填充） | 一开始不用 CHAR 存 ID | 已修复；无需回退 |
| **JWT 载荷** | `sub` + `uname`；可选「sub 缺失按用户名回查」（**默认关**） | 生产仅以 `sub` 为准；短 TTL；Refresh；吊销表/jti | **保持默认关**；若要吊销再加组件 |
| **JWT 密钥** | 默认 `VAGENT_SECURITY_JWT_SECRET` 可带**本地回退**；`prod` **无回退** | 仅环境变量/密钥管理服务 | **`application-prod.yml` 已强制 env** |
| **DDL 来源** | Flyway（PG）+ `schema-*.sql`（H2/M2）双轨；测试库直接用 **uuid 列** | 单轨或 codegen | 可接受 |
| **CORS** | `vagent.security.cors.allowed-origin-patterns`；默认 `*` | `application-prod.yml` 白名单域名 | **已配置化**；上线前改 `CHANGE_ME` |
| **数据源** | `docker-compose` 与默认 **`jdbc:.../vagent`**、`postgres/postgres` 对齐 | K8s 示例同库名 | **已对齐** |
| **MCP 参数** | `sanitizeToolArguments` 仅精细处理 `ping`/`echo` | 每工具 JSON Schema 校验 + 拒绝未知键 | 工具变多时必须加强 |
| **K8s 清单** | 单副本、无 Ingress、无 PVC 说明 | HA、持久化、NetworkPolicy、资源限制 | 标注「演示」即可；真上云再扩 |

## 3. 建议决策顺序（轻量 → 重）

1. **必做（若对外部署）**：JWT secret 外置、CORS 白名单、Actuator 暴露面复查。  
2. **低成本对齐**：compose / README 与默认库名一致；生产 profile 模板（`application-prod.example.yml`）。  
3. **中期**：主键与 JWT 统一为 **PG `UUID` + 应用层一致类型**（独立迁移版本）。  
4. **随产品**：意图/改写模型化、记忆摘要、MCP 强 schema、观测与 CI 里的 PG 集成测。
