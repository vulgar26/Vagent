## 2. 项目概述

### 2.1 项目名称与定位

| 项 | 内容 |
|----|------|
| **项目名称** | Vagent（Working title，可调整） |
| **一句话定位** | 基于 Java 的 **企业风格对话式 RAG 系统**，**独立编码实现**；主链路 **行为与阶段划分** 对齐下文 **§3 参考实现** 的拆解。 |
| **技术栈（规划）** | Java 17+、Spring Boot、关系型数据库、向量检索、SSE 流式、（按需）消息队列与独立工具服务等。 |

### 2.2 背景与动机

- 通过 **从 0 到 1 实现** 完整对话 RAG 主链路，掌握记忆、改写、意图、检索、Prompt 组装、流式生成与取消等工程能力。  
- 学习过程中以开源项目 **Ragent**（见 §3）为 **可读源码的参考实现**，弄清每一阶段职责与分支条件；**Vagent 代码不拷贝其源码**，类名与模块划分可不同，但 **对外可观察行为**（阶段顺序、主要分支）与 §3 一致，便于自测与验收。  
- 产出可维护、可测试、可演示的个人/作品集项目。

### 2.3 边界与诚信

| 项 | 说明 |
|----|------|
| **与参考实现的关系** | §3 描述的是 **Ragent 当前实现** 的能力拆解，作为 Vagent 的 **规格对照**；实现方式为自研。 |
| **诚信** | 若引用第三方代码片段，在文件头或 `NOTICE` 中按许可证注明；不整仓库未授权复制。 |
| **非目标（初版可不做）** | 与 Ragent **全仓库所有模块**（管理端、入库 Pipeline、MQ 等）逐一对齐；商业级 SLA 可在后续版本定义。 |

## 3. 参考实现：Ragent 主链路做了什么

本节说明 **Ragent**（仓库：[nageoffer/ragent](https://github.com/nageoffer/ragent)）在 **单次流式对话** 中完成的工作，作为 Vagent **必须覆盖的主链路规格**。代码锚点以 **当前主流实现** 为准：`bootstrap` 模块中 `RAGChatServiceImpl#streamChat`（入口控制器为 `RAGChatController`，路径示例：`GET /rag/v3/chat`，SSE）。

### 3.1 总体顺序（一句话）

**初始化会话与回调 → 加载记忆 → 改写/拆分 → 意图解析 → 歧义引导（可提前结束）→ 判断是否仅需系统提示（可不经检索直接流式）→ 否则检索（知识库 + 工具侧结果汇入同一上下文）→ 空检索则提示并结束 → 非空则组装 Prompt → 流式调用 LLM → 绑定任务句柄以支持取消。**

### 3.2 分阶段说明（与 Ragent 实现对应）

#### （1）入口与准备

| 步骤 | Ragent 在做什么 |
|------|-------------------|
| 会话与任务 ID | 确定 `conversationId`（无则生成）；确定 `taskId`（与 Trace 上下文配合，用于取消与追踪）。 |
| 流式回调 | 构造 `StreamCallback`（由 `StreamCallbackFactory` 创建，内部多为 `StreamChatEventHandler`），后续模型输出、完成、错误均经此回调写 SSE。 |
| 用户 | 从 `UserContext`（登录态）取 `userId`。 |
| 记忆 | `ConversationMemoryService#loadAndAppend`：按会话与用户加载历史（含摘要策略控制 token），将 **本轮用户消息** 追加进列表，得到 `history`。 |

#### （2）检索之前的「理解」

| 步骤 | Ragent 在做什么 |
|------|-------------------|
| 改写与拆分 | `QueryRewriteService#rewriteWithSplit`：输出 `RewriteResult`，可含 **子问题**；内部可按配置走规则或调用 LLM 改写/拆分，失败时有兜底。 |
| 意图解析 | `IntentResolver#resolve`：对每个子问题产出 `SubQuestionIntent`（含意图节点及得分等）。 |
| 歧义与引导 | `IntentGuidanceService#detectAmbiguity`：若置信不足等需 **向用户澄清**，则通过回调 **直接流式输出引导文案** 并结束，**不进行检索与主回答生成**。 |

#### （3）分支：是否「仅系统意图」（不经知识库/工具检索）

| 条件 | Ragent 在做什么 |
|------|-------------------|
| 所有子意图均为「系统类」 | `IntentResolver` 判断 `allSystemOnly`：此时 **不调检索**。使用 `streamSystemResponse`：组装系统提示（可来自意图节点模板或默认模板）+ 历史 + 用户问题，**直接** `LLMService#streamChat`，绑定 `taskId` 与取消句柄后结束。 |

#### （4）RAG 主路径：检索与生成

| 步骤 | Ragent 在做什么 |
|------|-------------------|
| 检索 | `RetrievalEngine#retrieve(subIntents, topK)` → `RetrievalContext`：聚合 **知识库检索片段** 与 **MCP 等工具调用结果**（具体通道、多路合并、后处理在检索子系统内）。 |
| 检索为空 | 固定提示（如未检索到相关文档），回调输出后结束，**不再调用大模型**。 |
| 检索非空 | `IntentResolver#mergeIntentGroup` 等合并意图信息，供 Prompt 侧区分 KB / 工具意图等。 |
| 拼 Prompt | `RAGPromptService#buildStructuredMessages`：将 KB 文本、工具上下文、意图 chunk 等写入消息列表；构造 `ChatRequest`（含 **深度思考** 开关；若存在 MCP 上下文，温度/topP 与纯 KB 场景 **参数不同**）。 |
| 流式生成 | `LLMService#streamChat`（实现侧含路由、多模型、流式首包探测等，属 `infra-ai`）。 |
| 取消 | `StreamTaskManager#bindHandle(taskId, handle)`：`stopTask(taskId)` 时可取消流与清理。 |

### 3.3 分支总览（与 Ragent 行为一致）

```
记忆加载
  → 改写/拆分
  → 意图解析
  → 需澄清？ ──是──→ 流式输出引导 → 结束
  → 否
  → 全系统意图？ ──是──→ 流式系统对话（不经检索）→ 结束
  → 否
  → 检索（KB + 工具侧上下文）
  → 空？ ──是──→ 提示无文档 → 结束
  → 否
  → 拼 Prompt → 流式 LLM → 绑定取消句柄
```

### 3.4 Vagent 实现时建议对齐的抽象（不限类名）

- **编排层** 保持与上表相同的 **阶段顺序与分支条件**；DTO 可改名，但职责需覆盖：`RewriteResult`、`SubQuestionIntent`、`RetrievalContext`、`StreamCallback`、`ChatRequest` 等。  
- **检索子系统** 内部可迭代（多通道、后处理、工具协议），但对外 **`retrieve(subIntents, topK)`** 作为编排入口宜稳定。  
- 若 Vagent 某阶段 **刻意简化**（例如先不接 MCP），须在 `DECISIONS.md` 写明与 §3 的差异及后续计划。

## 4. 目标与成功标准

### 4.1 总体目标

实现 **§3 所描述** 的主链路（含分支），并具备：**可运行、可观测、可测试、可按 README 部署**。

### 4.2 主链路范围（必须）

与 **§3** 一致，须覆盖：记忆、改写/拆分、意图、引导提前返回、系统意图直出、检索、空检索、拼 Prompt、流式 LLM、任务取消。

### 4.3 成功标准（验收）

| 编号 | 标准 | 验证方式 |
|------|------|----------|
| S1 | 登录用户可发起 **SSE 流式对话** | 浏览器/客户端演示 |
| S2 | 主链路 **阶段与分支** 与 **§3** 一致 | 代码走查 + 集成测试 |
| S3 | 检索为空时有明确提示，不胡乱生成 | 手工用例 |
| S4 | 支持 **中断** 流式生成（等价 `stopTask`） | 手工或自动化 |
| S5 | 核心纯逻辑具备 **单元测试**（指标自填） | `mvn test` |
| S6 | README：**启动步骤、依赖、配置项** | 他人可按文档跑通 |

## 5. 范围说明（Scope）

### 5.1 第一版（MVP）建议包含

- 单可部署应用或最小多模块  
- 一种向量存储方案（如 pgvector / Milvus 二选一）  
- 一种大模型接入（流式 + 取消）  
- 最小鉴权与用户隔离（若多用户）  

### 5.2 可延后（Backlog）

- 与 Ragent **同体量** 的管理端、完整 Trace 落库、多厂商路由、流式首包换模等 → `DECISIONS.md` 标记 Phase 2+。  

### 5.3 需求变更

影响主链路的变更须同步更新 **§3 / §4**，并在修订记录中说明。

## 6. 技术方案概要

### 6.1 逻辑架构

- **接入层**：HTTP（SSE）、鉴权、限流（可选）  
- **对话编排**：单入口 Service，顺序同 §3  
- **检索子系统**：可扩展检索 + 多通道/后处理（迭代）  
- **基础设施**：模型客户端、向量客户端、配置、日志与 Trace ID  

### 6.2 数据与部署

- 关系库：用户、会话、消息、（可选）意图与任务  
- 向量库：分块与索引策略见 `DECISIONS.md`  
- 部署：Docker Compose 或等价  

### 6.3 关键非功能需求

| 类别 | 要求 |
|------|------|
| 可观测 | 请求 ID、主链路各阶段耗时 |
| 安全 | 密钥环境变量注入；检索带租户/用户隔离条件 |
| 可维护 | 模块边界清晰；重大取舍有文档 |
