# M5 实现说明（改写、意图、澄清分支）

本文档说明 **M5 里程碑**：在 M4 RAG 主链路上增加 **检索前 query 改写** 与 **规则意图分支**（寒暄不经检索、过短澄清提前返回），与策划书 §3 中「改写 / 意图 / 歧义引导」的最小子集对齐；**未**实现 LLM 改写、子问题拆分、MCP 工具检索等，后续可替换接口实现。

---

## 1. M5 要达成什么

| 能力 | 说明 |
|------|------|
| 改写 | `QueryRewriteService#rewriteForRetrieval`：默认 **透传** 本轮用户句；可选 **拼接历史 USER** 再作为 `KnowledgeRetrieveService#search` 的 query。 |
| 意图 | `IntentResolutionService#resolve`：当前为 **规则实现**（前缀寒暄 → 过短澄清 → 否则 RAG）。 |
| 分支 | `ChatBranch`：`RAG` / `SYSTEM_DIALOG` / `CLARIFICATION`。 |
| SSE | `meta` 增加 **`branch`**（枚举名），`hitCount` 语义与 M4 一致；澄清分支仅 `chunk`+`done`，**不调** `LlmClient`。 |
| 开关 | `vagent.orchestration.intent-enabled=false` 时意图恒为 RAG，行为与纯 M4 一致（改写仍可按策略生效）。 |

---

## 2. 推荐阅读顺序

1. `OrchestrationProperties`（`application.yml` 中 `vagent.orchestration.*`）。  
2. `ConfigurableQueryRewriteService`、`RuleBasedIntentResolutionService`。  
3. `RagStreamChatService`（落库用户后：改写 → 意图 → 检索或直出）。  
4. `ChatStreamingConfiguration`（注册 `OrchestrationProperties`）。  
5. 单测：`ConfigurableQueryRewriteServiceTest`、`RuleBasedIntentResolutionServiceTest`。

---

## 3. 与策划书 §3 的差异（刻意简化）

| §3 能力 | Vagent M5 |
|---------|-----------|
| LLM 改写/拆分子问题 | 未做；仅字符串拼接或透传。 |
| 意图分类器 / 得分 | 未做；前缀 + 长度规则。 |
| 检索为空不调 LLM | 仍为 M4 行为：空命中仍调 LLM（SYSTEM 说明无片段）；可在后续里程碑加开关对齐 Ragent。 |
| 全系统意图 | 对应 `SYSTEM_DIALOG`：不调检索，仍调 LLM。 |
| 澄清 | 对应 `CLARIFICATION`：固定模板 SSE，不调 LLM。 |

---

## 4. 配置示例

```yaml
vagent:
  orchestration:
    intent-enabled: true
    rewrite-strategy: concat-user-history   # 或 passthrough
    concat-max-user-segments: 3
    clarification-min-chars: 3
    system-dialog-prefixes: "你好,嗨,hi,hello,谢谢,再见"
    clarification-template: "请补充…"
```

---

## 5. 自测建议

- **寒暄**：`你好` → `meta.branch=SYSTEM_DIALOG`，`hitCount=0`，仍有模型流式（`noop` 则无 chunk）。  
- **过短**：单字/少于 `clarification-min-chars`（且非寒暄前缀）→ `CLARIFICATION`，仅一条引导 `chunk`。  
- **拼接检索**：`rewrite-strategy: concat-user-history`，多轮后追问「同上」类问题，观察召回是否改善（需真实 PG + 向量数据）。
