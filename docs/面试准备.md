## 一、项目定位与目标

我参与实现的是 **Vagent**，一个面向**企业风格对话式 RAG** 的后端服务，技术栈是 **Java 17+、Spring Boot 3.x**。目标不是一次性做完全部 RAG 细节，而是按里程碑**可运行、可测试、可演进**：先搭好 Web、健康检查、可替换的 LLM 抽象；再做**用户与会话**；再做**知识库与向量检索**，为后续「改写、意图、编排、SSE」留好边界。

---

## 二、整体框架：分层与模块划分

在框架层面，采用的是典型的 **Spring Boot 分层 + 领域分包**，而不是「所有逻辑堆在 Controller」。

- **接入层（REST）**：`Controller` 只负责 HTTP 协议、参数校验、从安全上下文取当前用户，业务规则下沉到 Service。
- **领域/应用层（Service）**：编排用例，例如注册登录、建会话、知识库入库与检索；事务边界放在这里（`@Transactional`）。
- **持久化层**：**MyBatis-Plus** 做 CRUD 与简单条件查询；对 **PostgreSQL 特有类型**（如 `vector`）用 **自定义 TypeHandler** 或 Mapper 里显式 SQL。
- **横切能力**：**Spring Security + JWT** 做无状态认证；**统一异常与 JSON 响应**（如全局异常处理）；配置用 **`@ConfigurationProperties`** 分域绑定（JWT、LLM、Embedding 等）。

包结构上是 **按业务域划分**（`auth`、`user`、`conversation`、`security`、`llm`、`embedding`、`kb`），而不是按「dao/service/impl」单层大包，这样后续加「消息、检索编排、工具调用」时，依赖关系更清晰。

## 三、核心设计决策（为什么这样设计）

### 1. 数据存储：PostgreSQL 作为唯一业务库，向量同库

把**业务关系数据**和 **pgvector 向量数据**放在**同一个 PostgreSQL 实例、同一套连接配置**里。理由是：用户、会话、知识库文档、分块天然有外键与一致性需求；若 M2 就拆出独立向量库，会引入同步、双写、分布式事务或最终一致等复杂度，不符合当前里程碑「先跑通主链路」的目标。

### 2. DDL 拆分：`schema-core` 与 `schema-vector`

把建表脚本拆成 **`schema-core.sql`**（用户、会话等，与 H2 的 PostgreSQL 模式兼容）和 **`schema-vector.sql`**（`CREATE EXTENSION vector`、KB 表）。这样 **测试环境**可以继续用 **H2 内存库**跑 M1 回归，而不去执行 pgvector 专有语法；**开发/集成测试**用真实 PostgreSQL + pgvector。这是**环境可组合**的设计，而不是一个 `schema.sql` 强行适配所有数据库。

### 3. 认证与安全：JWT + 资源级按用户隔离

安全模型是 **无状态 JWT**：登录/注册放行，其余 `/api/v1/**` 需带 `Authorization: Bearer`。业务表里用 **无连字符的 UUID 字符串**存用户主键（与 MyBatis-Plus 的 UUID 策略一致），所有会话、KB 数据都带 **`user_id`**，查询时**必须**带当前用户条件，避免横向越权。这是多租户/多用户系统里**最低限度**的数据隔离。

### 4. LLM 与 Embedding：接口 + 配置驱动实现

- **LLM**：定义 **`LlmClient`** 接口，默认 **`noop`** 实现，保证应用可启动、测试不依赖外网；后续按 `provider` 切换实现类。
- **Embedding（M2）**：同样定义 **`EmbeddingClient`**，默认 **`HashEmbeddingClient`**——本地确定性向量、无外部依赖，用于联调与结构验证；向量做 **L2 归一化**，与 pgvector 的 **余弦距离 `<=>`** 一致。真实环境可换成 HTTP/本地模型实现，**Controller/Service 尽量只依赖接口**。

### 5. 知识库领域模型

KB 拆成两层实体：**文档 `kb_documents`**（标题、用户、时间）和 **分块 `kb_chunks`**（归属文档、序号、正文、`embedding`）。这样一篇长文对应多行分块，检索命中的是**块**，同时能回溯 **`document_id`**，后续拼 Prompt 时结构清楚。

### 6. 检索实现路径

- **入库**：正文经 **`TextChunkingService`** 按配置做字符窗口 + 重叠切分；每块调用 **`EmbeddingClient`**，经 MyBatis 插入；**`vector` 列**通过 **`PgVectorFloatArrayTypeHandler`** 写入。
- **检索**：对查询句做 **同样的 embed**，在 **`KbChunkMapper`** 里写 **显式 SQL**：`WHERE user_id = ?`，按 **`<=>` 排序** `LIMIT topK`。把向量距离计算放在 Mapper，是为了**集中调优**（以后加索引、改算子）而不污染业务层。

### 7. API 形态

知识库暴露 **`POST /api/v1/kb/documents`**（入库）和 **`POST /api/v1/kb/retrieve`**（检索），与 M1 的 **`/api/v1/auth/*`、`/api/v1/conversations/*`** 风格一致，全部挂在 **`/api/v1`** 下，由同一套 JWT 保护。

## 四、具体实现要点

### 1. 启动与扫描

Spring Boot 启动类上使用 **`@MapperScan("com.vagent")`** 扫描所有 Mapper；排除默认的 **`UserDetailsServiceAutoConfiguration`**，避免生成内存用户，与 JWT 模型一致。

### 2. 配置绑定

- **`JwtProperties`**：`vagent.security.jwt.*`
- **`LlmProperties`** + **`LlmClientConfiguration`**：按 `vagent.llm.provider` 注册 `LlmClient`
- **`EmbeddingProperties`** + **`EmbeddingClientConfiguration`**：按 `vagent.embedding.provider` 注册 `EmbeddingClient`，并绑定维度、分块参数

这样**密钥、模型、向量维度**都不会写死在业务代码里，环境变量或 profile 可覆盖。

### 3. 持久化

- **实体**：MyBatis-Plus 注解映射表名、字段；`KbChunk` 对 **`embedding`** 指定 **`typeHandler`** 和 **`autoResultMap`**，以支持非标准 JDBC 类型。
- **Mapper**：文档与分块用 **`BaseMapper`** 做 insert；检索用 **自定义 `@Select`**，返回 DTO（如命中项 + **distance**）。

### 4. 初始化数据脚本

主配置里 **`spring.sql.init`** 依次加载 **core + vector** 脚本；**test profile** 只加载 **core**，保证 `mvn test` 不依赖本机 PostgreSQL。生产环境应迁移到 **Flyway/Liquibase** 并关闭 `always`，这是文档里会明确写的演进方向。

### 5. 测试策略

- **默认单元/集成测试**：H2 + core schema，覆盖认证与会话主路径；对 hash 嵌入可做**不依赖 DB**的断言（维度、范数）。
- **真实 pgvector 链路**：用 **Testcontainers** 起带 pgvector 的 PostgreSQL 镜像，通过 **`@DynamicPropertySource`** 注入数据源；由于镜像拉取耗时，可在 **Surefire 里默认排除**该类，需要时单独执行，避免拖慢日常 `mvn test`。

## 五、当前边界与可扩展性

刻意没有在 M2 做：**真实语义 embedding**、**流式对话**、**检索重排**、**ANN 索引调优**、**按 token 的智能分块**。当前顺序扫描 + 小数据量足够验证架构；数据量上来后，在 **同一 Mapper/SQL 层**加 **HNSW/IVFFLAT** 并配合 `ANALYZE` 即可，不需要推翻分层。

---

## 六、30 秒口述小结

用 Spring Boot 搭分层架构，JWT 做多用户隔离，PostgreSQL 单库存业务与向量，MyBatis-Plus 持久化，向量列用 TypeHandler，检索 SQL 显式写 pgvector 距离；LLM 与 Embedding 都接口化、配置切换实现；DDL 按环境拆分以兼顾 H2 测试与 pgvector 真库；测试上默认轻量、向量全链路用 Testcontainers 可选跑。

## 七、追问与答法（扩展）

以下为常见追问方向，可随项目演进继续补充条目。

### 1. 为什么用 MyBatis-Plus 而不是 Spring Data JPA？

- 项目已选 **MyBatis-Plus**，SQL 可控、对 **PostgreSQL 特有类型**（`vector`）和复杂检索语句更直接；JPA 对自定义类型与原生向量运算通常要更多 **AttributeConverter / 原生查询** 样板代码。
- 团队若强偏好 JPA，也可行，但 **pgvector 检索** 仍会以 **原生 SQL/自定义函数** 为主，ORM 的收益有限。

### 2. 向量维度从 128 改成 256 要怎么迁移？

- **配置**：`vagent.embedding.dimensions` 与模型输出维度一致。
- **库表**：`kb_chunks.embedding` 的列类型要从 `vector(128)` 改为 `vector(256)`（或新列 + 回填），属于**破坏性变更**；已有向量在维度变化后**不能直接用**，必须 **重新嵌入** 全量分块。
- **流程**：Flyway/Liquibase 做 DDL；离线或异步任务重算 embedding；检索 SQL 里 `CAST(... AS vector(N))` 与列定义对齐。

### 3. JWT 无状态，「登出」和黑名单怎么做？

- **纯 JWT**：服务端不存会话，**登出**仅客户端删 token；无法吊销已签发 token。
- **加强**：短期 access token + refresh、**服务端存 refresh 版本号/会话表**；或引入 **Redis 黑名单**（jti + 过期时间）；或对敏感接口改用 **opaque session**。
- 口述时说明：**当前里程碑是 API 与数据隔离**，登出增强是**明确可演进点**。

### 4. 为什么检索 SQL 写在 Mapper 而不是 Service 里拼？

- **距离计算、排序、LIMIT、索引提示** 属于数据访问与性能调优，和 **领域规则**（例如「只查本用户」）在 Mapper 用 `WHERE user_id` 表达即可；Service 保持「嵌入 → 调查询」两步，**换索引、换算子**时改一处 SQL。

### 5. Hash 嵌入没有语义，上线怎么办？

- **替换实现**：新 `EmbeddingClient` Bean，配置切换 `provider`；维度与度量（余弦）与库表一致。
- **数据**：旧向量与新版模型不兼容时需 **重嵌入**；可双写、灰度或按文档版本迁移。

### 6. 多租户（tenant）与当前 `user_id` 隔离的区别？

- 当前是 **单租户产品内多用户**，用 `user_id` 隔离即可。
- 若 **SaaS 多租户**：表增加 `tenant_id`，JWT 带租户声明，**所有 SQL 带 tenant 条件**；向量索引与 RLS（行级安全）可进一步讨论。

### 7. Testcontainers 默认排除集成测试，会不会漏测？

- **权衡**：默认 `mvn test` 快速、不依赖 Docker；CI 可增加 **单独 job** 跑 `M2KnowledgeVectorIntegrationTest`。
- 口述：**本地与 CI 分层**：快速反馈 + 夜间/合并前全链路。
